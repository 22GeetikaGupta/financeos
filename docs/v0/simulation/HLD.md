# Simulation Engine

# High Level Design (HLD)

Version: 1.0

Status: Draft

Author: Geetika Gupta

---

# 1. Introduction

The Simulation Engine is the core orchestration component of FinanceOS.

Its primary responsibility is to coordinate the execution of all financial engines and produce a complete simulation result.

The Simulation Engine itself does not perform financial calculations.

Instead, it delegates financial calculations to specialized engines while controlling the overall execution lifecycle.

For Version 1, the Simulation Engine orchestrates only the Loan Engine.

Additional engines will be integrated in future versions.

---

# 2. Purpose

The purpose of the Simulation Engine is to provide a consistent execution framework for financial simulations.

It is responsible for:

- Receiving simulation requests
- Initializing simulation execution
- Invoking financial engines
- Managing execution order
- Aggregating results
- Returning simulation outputs

---

# 3. Responsibilities

The Simulation Engine is responsible for:

- Simulation initialization
- Input validation coordination
- Engine orchestration
- Execution sequencing
- Result aggregation
- Error propagation
- Simulation completion

The Simulation Engine is NOT responsible for financial calculations.

---

# 4. Scope

Version 1 supports:

- Home Loan simulation

Future versions will support:

- FD
- RD
- Mutual Funds
- Cash Flow
- Net Worth
- Scenario Comparison
- Optimization

---

# 5. High Level Architecture

```
                Simulation Request
                        │
                        ▼
               Simulation Engine
                        │
                Execution Controller
                        │
                        ▼
                  Loan Engine
                        │
                        ▼
                Simulation Result
                        │
                        ▼
                 Backend Response
```

The Simulation Engine acts as the orchestration layer between the API layer and the financial engines.

---

# 6. Components

The Simulation Engine consists of the following logical components.

## Request Handler

Receives simulation requests from the Backend API.

Responsibilities:

- Accept request
- Trigger simulation
- Return response

---

## Execution Controller

Coordinates execution of all financial engines.

Responsibilities:

- Determine execution order
- Invoke engines
- Collect outputs

Version 1 invokes only the Loan Engine.

---

## Result Aggregator

Collects outputs from all engines and prepares the final simulation response.

Responsibilities:

- Aggregate engine outputs
- Generate simulation summary
- Prepare response object

---

# 7. Simulation Lifecycle

Every simulation follows the same lifecycle.

```
Receive Request

↓

Validate Input

↓

Initialize Simulation

↓

Execute Financial Engines

↓

Aggregate Results

↓

Return Response
```

This lifecycle remains consistent even as new financial engines are added.

---

# 8. Request Flow

Step 1

Backend API receives simulation request.

↓

Step 2

Simulation Engine initializes execution.

↓

Step 3

Execution Controller invokes Loan Engine.

↓

Step 4

Loan Engine returns simulation data.

↓

Step 5

Result Aggregator prepares final response.

↓

Step 6

Backend API returns simulation result.

---

# 9. Engine Communication

The Simulation Engine communicates with financial engines through well-defined interfaces.

The orchestration flow is:

```
Simulation Engine

↓

Loan Engine

↓

Return Loan Results

↓

Simulation Engine

↓

Aggregate Results

↓

Response
```

Each financial engine remains independent and unaware of other engines.

This design minimizes coupling and simplifies future expansion.

---

# 10. Outputs

The Simulation Engine returns a unified simulation response.

Version 1 includes:

- Loan Summary
- EMI Information
- Loan Closure Date
- Total Interest Paid

Future versions will enrich this response with investment, cash flow, and net worth information.

---

# 11. Error Handling

The Simulation Engine is responsible for propagating execution failures.

Examples include:

- Invalid simulation request
- Engine execution failure
- Unsupported financial configuration

Errors generated by child engines are propagated back to the Backend API with appropriate context.

---

# 12. Future Evolution

As FinanceOS evolves, additional financial engines will be integrated into the execution flow.

Future execution flow:

```
Simulation Engine
        │
        ├── Loan Engine
        ├── Investment Engine
        ├── Cash Flow Engine
        └── Net Worth Engine
```

The Simulation Engine architecture is intentionally designed to support incremental addition of new financial engines without requiring changes to existing engine implementations.

---

# Summary

The Simulation Engine is the orchestration layer of FinanceOS.

It coordinates execution, manages simulation flow, and aggregates results while delegating all financial calculations to specialized engines.

This separation of orchestration and business logic ensures that the architecture remains modular, maintainable, and extensible as FinanceOS grows.
